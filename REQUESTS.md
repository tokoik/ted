# 作業依頼履歴

この文書は、プログラムの改善に関する依頼と、その依頼に基づいて実施した変更を記録します。

## 2026-07-21: `Menu.cpp` の構造点検

### 依頼

度重なる修正とクラスのスコープ制限によって複雑化した `Menu.cpp` を点検し、問題と改善方法を提案する。

### 点検結果

- Native File Dialog が返すパスの解放漏れ
- 設定再読み込み中に旧描画資源と新設定が混在する可能性
- UI 内に OpenXR、Leap Motion、投影更新などの副作用が分散
- 必須の `GgApp` を nullable なポインタとして保持
- `Window` と `Attitude` の状態を UI が直接編集
- カメラ能力列挙で Media Foundation 固有型が UI 層へ露出
- GUID からコーデック名への変換処理の重複
- 起動時設定保存、文字列入力バッファ、未使用変数などの整理候補

## 2026-07-21: 指摘 1 — ファイルパスの解放

- `getFilePath()` が取得したパスを `std::string` へコピーした後、`NFD_FreePath()` で解放するよう修正した。

## 2026-07-21: 指摘 2 — 設定再読み込みのトランザクション化

- 読み込んだ設定を `defaults` へ即時反映せず、`Menu` が候補設定として保持するよう変更した。
- `Rect` と `Scene` を候補設定から構築し、成功した場合だけ候補設定を確定するよう変更した。
- `Scene` の構築時に、設定ファイルのパスとシーングラフの深さ上限を候補設定から明示的に渡すよう変更した。
- 構築失敗時は候補設定を破棄し、実行中の設定を維持する。

## 2026-07-21: 指摘 3 — 副作用を操作 API へ集約

- 表示モード変更を `GgApp::Window::setDisplayMode()` へ集約した。
- クリップ面の検証、設定、投影更新を `GgApp::Window::setClipPlanes()` へ集約した。
- ハンドトラッキング方式の変更とLeap Motionの開始・停止を `GgApp::setHandTrackingMode()` へ集約した。
- ミラー表示とシーン表示をアクセサー経由で変更するようにした。
- `Menu` は UI 入力を取得し、操作 API を呼び出す役割に限定した。

## 2026-07-21: 指摘 4 — クラス境界の整理

- `Menu` が保持する `GgApp*` を `GgApp&` に変更し、null 状態を排除した。
- `Attitude` に姿勢調整値の取得・変更 API を追加し、`Menu` からの直接編集を廃止した。
- `Window` の `showScene`、`showMirror`、`showMenu` を非公開化し、描画ループを含めアクセサー経由へ変更した。

## 2026-07-21: 指摘 5 — カメラ能力 API の分離

- バックエンド非依存の `CaptureCapability` と `CameraCapabilities` API を追加した。
- Media Foundation の `VideoFormat` と `GUID` を `Menu` から排除した。
- GUID からコーデック名への変換を `CamMf.cpp` 側へ集約した。
- `Menu.h` から `CamMf.h` への直接依存を除去した。

## 2026-07-21: `GgApp` と TED 本体の分離検討

### 依頼

`GgApp.h`／`GgApp.cpp` から TED 本体への依存をなくせるか検討する。

### 方針

- GLFW、OpenGL、OpenXR、ImGui を扱う汎用ウィンドウ層と、`Config`、`Attitude`、`Scene`、`Camera` を所有する TED 本体を分離する。
- 現在の外側の `GgApp` は TED 本体として `TedApp` へ移し、`GgApp::Window` は独立した `GgWindow` とする案を基本とする。
- 入力処理は TED の状態を直接変更せず、イベントまたはコールバックとして TED 側へ通知する。
- OpenXR 層は追跡結果を返し、`Scene` への反映は TED 側で行う。
- この分離は影響範囲が大きいため、既存の `Menu` 改善と API 化を先に記録・安定化してから実施する。

## 2026-07-22: バグ修正と信頼性強化（第1弾〜第6弾）

### 依頼
繰り返し発生したエッジケースでの不整合、OpenXR API状態遷移違反、UDP周回判定の履歴、Leap Motionの点滅とデータレース等のバグを修正する。

### 修正内容

1. **入力切替失敗時の状態復元** (`ted.cpp`)
   - `useImage` などの入力切替で、デバイス・ファイルのオープンに完全に成功した後にのみ内部状態を更新し、失敗時は元の動作を維持するトランザクション構造へ修正しました。
2. **遅延UDPパケット混入の防止** (`Network.cpp`, `Network.h`)
   - `Packet` 構造体へ `frameId` を追加し、異なるIDの遅延パケットを破棄する順序制御を実装しました。
3. **姿勢テーブルの境界検証** (`Scene.cpp`)
   - `defaults.max_level` を超える深さの制限や、必要な配列サイズを満たさない不正な設定値を拒否する境界ガードを追加しました。
4. **受信画像デコードの競合防止** (`CamRemote.cpp`)
   - `image` サイズの読み出しをミューテックス保護下にし、`imdecode` のデコード失敗時（空画像）の安全ガードを追加しました。
5. **OpenXR初期化失敗時のロールバックとエラー分離** (`GgApp.cpp`)
   - `initOpenXR()` 専用の `XR_CHECK_INIT` マクロを導入し、初期化エラー時の確実なロールバックを実装しました。また、実行時 `xrEndFrame` のエラーハンドリングと分離しました。
6. **OpenXR画像取得失敗・トラッキング喪失時の描画スキップ** (`GgApp.cpp`)
   - 画像取得失敗やビュー姿勢が無効（トラッキング喪失）な際、レイヤーなしの空フレームを `xrEndFrame` に提出する安全なスキップ論理を実装しました。
7. **Leap Motionスナップショット化とスレッド同期** (`LeapListener.cpp`)
   - Leap同期用変数を `std::atomic` に変更し、SDK所有の手配列を内部バッファへディープコピーしたうえで、ロック中にローカルスナップショットを取得する構造へ変更しました。
8. **ダングリングポインタの防止** (`ted.cpp`)
   - 非ステレオ切替時に、破棄された旧カメラを指す `image[camR]` を確実に `nullptr` クリアするようにしました。
9. **手の点滅（フリッカー）防止と選択的消去** (`LeapListener.cpp`, `Scene.cpp`, `GgApp.cpp`)
   - 新しいフレームがない間は以前の姿勢を維持し、新しいフレームがあるときのみ、そのフレーム内に存在しない手だけを選択的にゼロクリアする論理を実装し、フリッカーを解消しました。また、OpenXRの手もトラッキング喪失時に即座にゼロクリア（消去）されます。
10. **タイムアウト付きUDP自動再同期と履歴更新の適正化** (`Network.cpp`, `Network.h`)
    - 正常に完了したフレームIDのみを `lastFrameId` に永続化して汚染を防ぎつつ、2.0秒以上の無通信タイムアウトで履歴を無効化し、送信側再起動によるIDリセットへ自動追従する論理を実装しました。
11. **起動時OpenXR失敗フォールバック** (`ted.cpp`)
    - 起動時の `startHMD()` 失敗時に、安全に表示モードを `MONOCULAR` に戻してビューポートを再設定するフォールバックを追加しました。
12. **安全なOpenXRフレーム終了とセッション強制破棄** (`GgApp.cpp`)
    - `commit()` での解放失敗時は `xrSessionStopPending` フラグを立て、`xrEndFrame` の呼び出し後に `stopHMD()` を実行する状態遷移へ変更しました。`xrEndFrame` 自体が失敗した場合もセッションを破棄して単眼表示へ復旧します。また、通常のクリーンアップでは `xrEndSession()` を呼ばず、任意の状態で呼べる `xrDestroySession()` を使用します。ランタイムから `STOPPING` が通知された通常の状態遷移では、引き続き `xrEndSession()` を呼びます。
13. **stopHMD後のミラー処理即時スキップ** (`GgApp.cpp`)
    - 遅延停止処理で `stopHMD()` が実行された直後に `return;` を追加し、解体済みのミラーFBOやスワップチェーンへ後続の `glBlitFramebuffer` 処理が進み `GL_INVALID_OPERATION` などのエラーが発生するのを防止しました。
